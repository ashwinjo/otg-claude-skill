Orchestration

6 Workflow Patterns

Claude classifies user intent and selects the right agent sequence automatically — no manual routing required.

Pattern 1 — Greenfield: Full Pipeline

deployment→ config→ script

Trigger: "Create BGP test + deploy + script". All three agents run sequentially. Port mapping from deployment feeds into config; config feeds into script.

Pattern 2 — Existing Infrastructure

config→ script

Trigger: "I have ixia-c at localhost:8443". Skip deployment. Provide known port locations directly to the config generator.

Pattern 3 — Parallel: Deploy + Licensing

deployment ∥ parallel ∥ licensing → config→ script

Deployment and licensing run simultaneously, then reconverge at config generation. Full pipeline with cost estimate in parallel.

Pattern 4 — Licensing Only

licensing

Trigger: "What license for 4×100GE + 8 BGP?". Single agent, standalone cost estimation with tier comparison.

Pattern 5 — IxNetwork Migration

convert→ config→ script

Trigger: "Migrate IxNetwork config to OTG". Conversion skill outputs OTG JSON; feeds downstream into config generator, then script generator.

Pattern 6 — Config Only

config

Trigger: "Generate OTG config from this description". Single agent produces OTG JSON only — no deployment, no script.

⚠ Critical: Port Alignment — After deployment, always extract port_mapping and inject it into the config generator. Misalignment causes HTTP 400 "location not found" at runtime.

Location format by deployment type:
Docker Compose (host network): veth-a, veth-z  ·  Docker Compose (port mapping): localhost:5555  ·  Containerlab: eth1, eth2

New · Institutional Memory

Per-Skill Learning System

Every mistake becomes permanent institutional knowledge. Each skill owns a fixes.md — agents read it at invocation, closing the learning loop automatically.

Architecture: Each skill's SKILL.md contains a fixes.md pointer. When a skill is invoked, the agent reads the co-located fixes.md before generating output. Errors discovered during execution are added back — mistakes become prevention rules.
.claude/skills/*/fixes.md — per-skill learning logs

Design

Architecture Highlights

Four design pillars that make the plugin robust, composable, and self-improving.

Skill-Based Architecture

Skills, not traditional code modules.

  • Self-contained — logic, references, evals, fixes
  • Composable — invoke independently or chain
  • SKILL.md + README.md + fixes.md per skill
  • evals.json test cases (4–8 scenarios each)

Subagent Orchestration

Claude Code as orchestrator, not executor.

  • Parse intent → classify workflow pattern
  • Dispatch to specialized agents
  • Data handoff (port_mapping across agents)
  • Parallel execution for independent tasks

Command-Driven Interface

16 Claude Commands across three tiers.

  • C — Discovery & help (7 commands)
  • A — Skill shortcuts with defaults (5 commands)
  • B — End-to-end workflow orchestration (4 commands)
  • ~150 KB total command documentation

Verified Artifacts Library

Permanent catalog of working configs and scripts.

  • Agents check catalog before generating
  • Reuse or regenerate — user decides
  • Survives all /kengotg-cleanup runs
  • INDEX.md with metadata per folder
