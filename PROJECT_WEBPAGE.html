Project Overview

A comprehensive network test automation toolkit that converts IxNetwork configs to OTG format, generates traffic test configurations from natural language, creates executable test scripts, deploys Ixia-c infrastructure, and recommends licensing strategies.

Five Production-Ready Skills

🔄 ixnetwork-to-keng-converter

Migrate IxNetwork configurations to OTG format with feasibility analysis.

Detects unsupported features, generates OTG JSON, provides conversion report.

🎯 otg-config-generator

Generate OTG configurations from natural language test descriptions.

BGP, ISIS, LACP, LLDP protocols. Validates against OTG schema.

🐍 snappi-script-generator

Create standalone Python Snappi test scripts from OTG configs.

Self-contained, production-ready. Metrics, assertions, error handling.

🚀 ixia-c-deployment

Deploy Ixia-c infrastructure (Docker Compose, Containerlab).

Manages veth pairs, controller health, port discovery.

💰 keng-licensing

Recommend licensing tiers and calculate costs (DPLU/CPLU).

Developer, Team, System. Per-port and per-protocol pricing.

Four Intelligent Subagents

🔵 ixia-c-deployment-agent

Infrastructure Provisioner — Analyzes requirements, generates Docker/Containerlab manifests, manages veth setup, validates health.

🟢 otg-config-generator-agent

Intent Translator — Parses test scenarios, injects port locations, generates OTG JSON, validates against schema.

🟣 snappi-script-generator-agent

Script Executor — Converts OTG configs to Python scripts, embeds infrastructure details, adds error handling & reporting.

🟠 keng-licensing-agent

Licensing Advisor — Calculates DPLU/CPLU costs, recommends tiers, provides cost breakdowns with SE disclaimers.

Intelligent Agents (4)

🔵 ixia-c-deployment-agent

Role: Infrastructure Provisioner

Responsibilities:
  • Analyze deployment requirements (Docker Compose vs Containerlab)
  • Generate infrastructure manifests (docker-compose.yml, topo.clab.yml)
  • Configure veth pairs, network namespaces, controller locations
  • Validate controller health and port connectivity
  • Return port mapping for downstream agents
Success Criteria:
  • Controller reachable and healthy
  • All ports discoverable
  • Port mapping in standardized format
Invoke:
@ixia-c-deployment-agent Deploy Ixia-c with Docker Compose for BGP testing

🟢 otg-config-generator-agent

Role: Intent Translator (Natural Language → OTG Config)

Responsibilities:
  • Parse test scenario descriptions
  • Validate protocol and feature support
  • Inject port locations from deployment agent
  • Generate standards-compliant OTG JSON
  • Align port names with infrastructure locations
  • Validate against OTG schema
Success Criteria:
  • OTG config validates against schema
  • All ports align with infrastructure
  • Device/flow/assertion definitions complete
  • Config is production-ready
Invoke:
@otg-config-generator-agent Create BGP test: 2 ports, AS 65001/65002, 1000 pps bidirectional

🟣 snappi-script-generator-agent

Role: Script Executor (OTG Config → Executable Python)

Responsibilities:
  • Generate standalone Python Snappi scripts
  • Embed OTG config and infrastructure details
  • Implement protocol setup, traffic control, state polling
  • Add assertions, metrics collection, error handling
  • Generate cleanup and JSON reporting logic
  • Validate script syntax and Snappi imports
Success Criteria:
  • Script executes without external config files
  • Error handling covers connection/timeout/protocol failures
  • Metrics and assertions properly integrated
  • JSON report generated with clear results
Invoke:
@snappi-script-generator-agent Generate Snappi script from bgp_convergence.json

🟠 keng-licensing-agent

Role: Licensing Advisor

Responsibilities:
  • Calculate data plane costs (KENG-DPLU) based on port speeds
  • Calculate control plane costs (KENG-CPLU) based on session count
  • Recommend license tier (Developer/Team/System)
  • Provide cost breakdown and tier comparison
  • Include Solutions Engineer disclaimers
  • Validate licensing assumptions
Success Criteria:
  • Cost calculations accurate (DPLU/CPLU formulas)
  • Recommendations match user requirements
  • Tier comparison is complete
  • SE disclaimer included
Invoke:
@keng-licensing-agent What license do I need for 4×100GE + 8 BGP sessions?

Orchestration Patterns

Pattern 1: Greenfield — Full Pipeline

Deploy → Generate Config → Generate Script

Use when: "Create BGP test + deploy + script"

Agent Sequence: ixia-c-deployment-agent → otg-config-generator-agent → snappi-script-generator-agent

All three agents run sequentially. Port mapping from deployment feeds into config generation. Config feeds into script generation.

Pattern 2: Existing Infrastructure

Generate Config → Generate Script

Use when: "I have ixia-c at localhost:8443; create test"

Agent Sequence: otg-config-generator-agent → snappi-script-generator-agent

Skip deployment. Provide known port locations directly to config generator.

Pattern 3: Parallel — Deploy + Licensing

Deploy + Licensing (parallel) → Config → Script

Use when: "Full test setup + license cost check"

Agent Sequence: ixia-c-deployment-agent ∥ keng-licensing-agent → otg-config-generator-agent → snappi-script-generator-agent

Both deployment and licensing run simultaneously, then reconverge at config generation.

Pattern 4: Licensing Only

Licensing (standalone)

Use when: "What license for 4×100GE + 8 BGP?"

Agent Sequence: keng-licensing-agent

Single agent, standalone cost estimation.

Pattern 5: Migration — IxNetwork → OTG

Convert → Generate Config → Generate Script

Use when: "Migrate IxNetwork config to OTG + run test"

Skill Sequence: ixnetwork-to-keng-converter → otg-config-generator → snappi-script-generator

Conversion skill outputs OTG JSON, which feeds into config generator, then script generator.

Pattern 6: Config Only

Generate Config (standalone)

Use when: "Generate OTG config from description"

Agent Sequence: otg-config-generator-agent

Single agent produces OTG JSON. No deployment, no script generation.

⚠️ Critical: Port Alignment

After deployment, ALWAYS extract port_mapping and inject into config generator.

Port locations vary by deployment type:
  • Docker Compose (host network): veth-a, veth-z (or custom veth names)
  • Docker Compose (port mapping): localhost:5555, localhost:5556
  • Containerlab: eth1, eth2 (or topology-defined names)

Misalignment → HTTP 400 "location not found" error.

Key Learning: fixes.md

The project includes a learning system: fixes.md documents all discovered issues, root causes, and solutions.

Future agents read fixes.md at session start and learn from past mistakes automatically.

Example: When the agent encounters a port location mismatch, it checks fixes.md, finds the lesson, and prevents the mistake.

Critical Architecture Rules (from fixes.md)

  • Never use ixia-c-one for Docker Compose — Only for Containerlab. Use separate containers for Compose deployments.
  • Port alignment is critical — After deployment, extract port_mapping and inject into config generator
  • API endpoint is /config — Not /api/v1/config. Controller at localhost:8443
  • Self-signed certificates require -k flag — curl -k for verification
  • Use continuous duration, never fixed_seconds — fixed_seconds crashes keng-controller
  • Snappi uses cfg.deserialize(), not cfg.loads() — loads() doesn't exist
  • Flow rate choice must be percentage, not line — OTG schema allows "line", Snappi rejects it

Architecture Highlights

1. Skill-Based Architecture

The project is organized around skills (`.claude/skills/`), not traditional code modules. Each skill is:

  • Self-contained: Owns its own logic, references, test cases
  • Composable: Can be invoked independently or chained with others
  • Documented: SKILL.md (technical), README.md (user guide), evals.json (test cases)

2. Subagent Orchestration Pattern

Claude Code acts as an orchestrator that:

  • Parses user intent
  • Dispatches to specialized agents
  • Manages data handoff between agents
  • Handles parallelism when appropriate

3. Command-Driven Interface (16 Commands)

Three tiers of Claude Commands:

  • Section C (7): Help & Discovery — Onboarding
  • Section A (5): Skill Shortcuts — Quick access with defaults
  • Section B (4): Workflows — End-to-end orchestration

4. Institutional Learning via fixes.md

Every mistake becomes institutional knowledge:

  • Documented in fixes.md (root cause, solution, prevention)
  • Promoted to skill SKILL.md (known limitations, patterns)
  • Agents read fixes.md at startup and learn automatically
